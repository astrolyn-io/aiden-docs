# Configuration

> 4 lines of YAML. That's all it takes to configure AIDEN. The rest is a bonus for perfectionists.

**What you'll learn:**
- The 5 configuration levels and how they nest together
- The available options to fine-tune every aspect
- Enterprise configuration to standardize across your teams

## The 5 levels: Russian nesting dolls

Think of Russian nesting dolls: each configuration level wraps the previous one. The smallest doll (AIDEN defaults) is always there. Each layer you add overrides it, without ever breaking what's underneath.

1. **AIDEN defaults** (built-in) -- the base, always present
2. **Enterprise** (`~/.aiden/enterprise.config.yaml`) -- your organization's standards
3. **Project** (`.aiden/aiden.config.yaml`) -- specific to this repo
4. **Feature** (`.aiden/features/<slug>/.override.yaml`) -- per-feature adjustments
5. **CLI flags** (`--mode`, `--provider`, etc.) -- one-off override, highest priority

Result: you only configure what changes. Everything else automatically inherits from the level above.

## Minimal configuration: 4 lines, let's go

You don't need to understand everything to get started. Create a file, paste 4 lines, and AIDEN works:

```yaml
# .aiden/aiden.config.yaml
project: my-project
provider: claude-code
mode: standard
lang: en
```

4 lines are enough to get started.

It's the outermost Russian doll for your project. It overrides AIDEN defaults with your choices. Anything you don't specify keeps its default value.

## Full options

When you're ready to go further, here's everything you can control:

| Key | Type | Default | Description |
|---|---|---|---|
| `project` | string | (required) | Project name |
| `provider` | string | `claude-code` | Default provider |
| `mode` | enum | `standard` | flash, standard, enterprise |
| `lang` | string | `fr` | Artifact language |
| `providers` | object | (built-in) | Per-provider config |
| `routing` | object | prefer | Routing strategy |
| `budget` | object | (built-in) | Token budget |
| `hooks` | object | {} | Event hooks |
| `rules` | array | [] | List of rule files |
| `pipeline` | object | (see below) | Inter-agent pipeline configuration |

### Pipeline

```yaml
pipeline:
  maxOutputCharsPerAgent: 8000  # Agent output truncation (default: 8000)
```

This option controls how much text from a previous agent is injected into the next agent's prompt. With a large-context LLM (200k+), you can increase this value. Reduce it if you're hitting token limits.

## Enterprise configuration: the innermost doll

Here's where things get interesting for teams. The enterprise config lives in your home directory and applies to **all** your AIDEN projects on the entire machine. It's the common foundation for your organization.

```yaml
# ~/.aiden/enterprise.config.yaml
company: my-company
defaults:
  mode: standard
  lang: en
  provider: claude-code
  require_approval: true
security:
  no_secrets_in_artefacts: true
  scan_dependencies: true
coding:
  test_coverage_min: 80
  commit_convention: conventional
```

A new dev joins the team? They install the enterprise config, and all AIDEN projects immediately follow the standards. No copy-pasting, no forgetting.

## Modify via CLI

Don't feel like editing YAML by hand? The CLI does it for you:

```bash
$ aiden config set mode flash
Set mode = flash

$ aiden config show
{ "project": "my-project", "mode": "flash", ... }
```
