# The 3 AIDEN modes: choose your weapon

> **TL;DR** — Bug fix → flash. Feature → standard. Critical migration → enterprise. When in doubt, standard.

Imagine three tools in front of you. A **Swiss army knife** for everyday emergencies. A **full workshop** for building something solid. And a **certified construction site** with blueprints, inspections, and sign-offs at every step. AIDEN works exactly like that — three modes tailored to the scope of your task.

This guide will show you how to choose the right mode, and more importantly, how AIDEN can do it for you.

## At a glance

| Feature | Flash | Standard | Enterprise |
|---|---|---|---|
| Time | < 30 min | 2h - 1 day | 1 day - 1 week |
| Agents | Lead, Dev | +Analyst, Planner, QA | +DocOps |
| Brief | No | Yes | Yes |
| Spec | No | Yes | Yes + ADR |
| Review | No | Yes | Yes + quality report |
| Human approval | No | No | Yes |

Let's look at each mode in detail.

## Flash — the Swiss army knife

Flash mode is pure responsiveness. Got a bug, a small refactoring, a quick tweak? Two agents (Lead + Dev), zero paperwork, and it's done.

```bash
$ aiden flash "Fix bug #123"
```

It's the right choice when:
- Less than 3 files affected
- No new API
- Low complexity (1-2/5)

Think of it as a 30-minute sprint: get in, fix it, get out.

## Standard — the full workshop

Now we're getting serious. Standard mode mobilizes five agents — analysis, planning, implementation, review — to produce a structured and tested result.

```bash
$ aiden standard "New users API"
```

It's the right choice when:
- 3 to 15 files affected
- New API or integration
- Medium complexity (3-4/5)

The majority of your features will go through this mode. It's the sweet spot between speed and rigor.

## Enterprise — the certified construction site

When the slightest misstep can be costly — data migration, GDPR compliance, architecture overhaul — Enterprise mode adds DocOps and, most importantly, **human approval between each phase**.

```bash
$ aiden enterprise "Authentication migration"
```

It's the right choice when:
- More than 15 files
- New architecture
- Compliance required (GDPR, security)

### Human approval: the safety net

In Enterprise mode, **each phase requires human approval** before proceeding to the next. At the end of each phase, the CLI displays:

```
[APPROVAL] Phase "discovery" for "migration-auth" completed.
  Agent: analyst | Tokens: 2340
  Preview: COMPLEXITY: 4 RISKS: - Data loss during migration...
  Approve? (y/n):
```

If you answer `n`, the workflow is aborted with the status `aborted`.

To bypass approvals (for example in CI/CD):

```bash
$ aiden enterprise "Auth migration" --auto-approve
```

> **Warning**: `--auto-approve` disables all human validation. Use only in automated pipelines.

## What if you don't know which mode to choose?

Good news: you don't have to. The Lead analyzes your feature description using heuristics and suggests the optimal mode:

- `"Fix typo in README"` → **flash** (low complexity, 1 file)
- `"Add new REST API with OAuth"` → **standard** (new API, external dependency)
- `"Migrate monolith to microservices with GDPR compliance"` → **enterprise** (new architecture, compliance)

The detection also takes project context into account (number of technologies, architecture). In short, the more complex your project, the more cautious AIDEN will be.

## Dynamic switching: shifting gears without stopping

Here's where it gets really clever. AIDEN can switch modes mid-flight **without losing work** and by executing the missing phases:

```bash
# Starts in flash, switches to standard if complexity detected
$ aiden flash "CSV export"
AIDEN Lead > Complexity detected (5 files). Switch to standard? [Y/n]
```

When you switch from flash to standard, `switchMode()`:
1. Identifies already completed phases (implementation)
2. Executes the missing phases (discovery, planning)
3. Continues the workflow with the new artifacts

Switching from standard to flash is a no-op — flash phases are a subset of standard.

It's like switching from a screwdriver to a power drill mid-project: you keep the hole already made, you just change the tool.
