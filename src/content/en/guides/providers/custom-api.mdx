# Custom API Provider

> Your company has an internal LLM proxy? Your team uses OpenAI? You have a GPU cluster with vLLM? AIDEN plugs into anything that speaks OpenAI-compatible.

> **What you'll learn**
> - Connect AIDEN to any LLM via OpenAI-compatible API
> - Configure LiteLLM, Ollama, Azure OpenAI and direct OpenAI
> - Set up an enterprise proxy with centralized logging
> - Run an on-premise LLM in an air-gapped environment

## How It Works

When the `baseUrl` field is set in a provider's configuration, AIDEN automatically switches to the **HTTP adapter** instead of the CLI subprocess. No external binary is required: AIDEN sends prompts via `fetch()` as `POST /chat/completions` in standard OpenAI format.

```
CLI Provider (default)        Custom API Provider
─────────────────────         ──────────────────────
AIDEN                         AIDEN
  └─ spawn(claude -p -)         └─ fetch(POST /chat/completions)
       ↑ stdin pipe                   ↑ JSON body (OpenAI format)
       claude CLI                     Your endpoint
```

The **provider factory** automatically detects the adapter type:

- Known name (`claude-code`, `gemini`, `copilot`, `opencode`) without `baseUrl` -> CLI adapter
- `baseUrl` set -> HTTP API adapter, regardless of the name
- `enabled: false` -> ignored

The `command` field must be empty (`command: ""`) for API providers to prevent any subprocess launch attempt.

## LiteLLM — Universal Proxy

[LiteLLM](https://github.com/BerriAI/litellm) is an open source proxy that exposes an OpenAI-compatible API in front of 100+ models (Anthropic, Google, OpenAI, Mistral, Cohere, local models...). It's the recommended solution for teams that want to centralize LLM access.

### Installation and Startup

```bash
$ pip install litellm
$ litellm --model claude-sonnet-4-6
# Proxy started on http://localhost:4000
```

To start with multiple models simultaneously, use a LiteLLM configuration file:

```bash
$ litellm --config litellm_config.yaml
```

### AIDEN Configuration

```yaml
# .aiden/aiden.config.yaml
provider: litellm

providers:
  litellm:
    enabled: true
    baseUrl: http://localhost:4000/v1
    model: claude-sonnet-4-6
    maxTokens: 100000
    costPer1k: 0.003
    command: ""
```

LiteLLM is particularly useful in enterprise settings for:
- Centralizing API keys (they stay on the proxy, not on each workstation)
- Logging all LLM calls in an observability tool (Langfuse, Helicone)
- Applying rate limits and quotas per team
- Switching models without changing the AIDEN config

## Ollama — Local LLM

[Ollama](https://ollama.com) lets you run open source LLMs locally (LLaMA 3, Mistral, Qwen, Phi, Gemma...). Ideal for air-gapped environments or for prototyping at zero cost.

### Installation and Model Download

```bash
$ curl -fsSL https://ollama.com/install.sh | sh
$ ollama pull llama3
# Downloads the model (~4 GB for llama3:8b)

$ ollama serve
# Starts the server on http://localhost:11434
```

Verify the server is accessible:

```bash
$ curl http://localhost:11434/v1/models
# Should return the list of downloaded models
```

### AIDEN Configuration

```yaml
providers:
  local-llm:
    enabled: true
    baseUrl: http://localhost:11434/v1
    model: llama3
    maxTokens: 8000
    costPer1k: 0
    command: ""
```

> **Note**: `costPer1k: 0` is correct for a local model — no external API call, no per-token cost.

### Choosing the Right Ollama Model

| Model | RAM required | Recommended use case |
|---|---|---|
| `llama3:8b` | 8 GB | Development, testing, prototyping |
| `llama3:70b` | 48 GB | Quality close to cloud models |
| `mistral:7b` | 6 GB | Fast code generation |
| `qwen2.5-coder:7b` | 6 GB | Code-specialized, good Dev results |
| `phi3:mini` | 4 GB | Machines with limited RAM |

```bash
$ ollama pull qwen2.5-coder:7b
```

```yaml
providers:
  local-llm:
    enabled: true
    baseUrl: http://localhost:11434/v1
    model: qwen2.5-coder:7b
    maxTokens: 32000
    costPer1k: 0
    command: ""
```

## OpenAI — Direct Connection

If your team uses OpenAI, you can connect AIDEN directly to the API without going through a proxy.

```yaml
providers:
  openai:
    enabled: true
    baseUrl: https://api.openai.com/v1
    apiKey: sk-...
    model: gpt-4o
    maxTokens: 128000
    costPer1k: 0.005
    command: ""
```

> **Warning**: Never commit your `apiKey` to the Git repository. Use an environment variable or add the `.aiden/aiden.config.yaml` file to `.gitignore`.

To inject the key via environment variable, most LLM proxies (LiteLLM, Cloudflare AI Gateway) accept `OPENAI_API_KEY` directly. On the AIDEN side, you can omit the key from the config and pass it via env.

### Choosing the OpenAI Model

| Model | Context | Strengths |
|---|---|---|
| `gpt-4o` | 128k | Best quality/speed/cost balance |
| `gpt-4o-mini` | 128k | Fast, excellent quality/price ratio |
| `o3` | 200k | Complex reasoning, architecture |
| `o4-mini` | 200k | Fast reasoning, good for QA |

```yaml
providers:
  openai:
    enabled: true
    baseUrl: https://api.openai.com/v1
    apiKey: sk-...
    model: gpt-4o-mini
    maxTokens: 128000
    costPer1k: 0.00015
    command: ""
```

## Azure OpenAI

Azure OpenAI exposes OpenAI models through your own Azure instance, with Microsoft's compliance, replication and network isolation guarantees.

```yaml
providers:
  azure:
    enabled: true
    baseUrl: https://mycompany.openai.azure.com/openai/deployments/gpt-4o
    apiKey: <your-azure-key>
    model: gpt-4o
    maxTokens: 128000
    costPer1k: 0.005
    command: ""
```

The Azure URL follows the format `https://<resource>.openai.azure.com/openai/deployments/<deployment-name>`. The deployment name is the one you created in the Azure AI Studio portal.

### Authentication via Environment Variable

```bash
$ export AZURE_OPENAI_API_KEY=<your-azure-key>
```

Then in the config, omit `apiKey` if your proxy or endpoint reads the variable automatically, or leave it with an explicit reference.

## Enterprise Proxy with Logging

This scenario covers the typical enterprise case: all AI requests go through an internal proxy that logs, rate-limits and manages credentials centrally. Developer workstations have no access to any API key.

### Architecture

```
Developer workstation      Internal proxy              Cloud LLM
──────────────────         ──────────────────         ──────────────
AIDEN (fetch)    ─────►   ai-proxy.corp/v1   ─────►  OpenAI / Anthropic
                           │
                           ├─ Logging (Langfuse)
                           ├─ Rate limiting per team
                           ├─ Content filtering
                           └─ API key rotation
```

### AIDEN Configuration

```yaml
# .aiden/aiden.config.yaml
provider: internal-proxy

providers:
  internal-proxy:
    enabled: true
    baseUrl: https://ai-proxy.corp.internal/v1
    apiKey: internal-token
    model: claude-sonnet-4-6
    maxTokens: 100000
    costPer1k: 0.003
    command: ""
```

The `apiKey` here is an internal token issued by your proxy — not an Anthropic or OpenAI API key exposed to developers.

### LiteLLM Configuration as Enterprise Proxy

Here's an example `litellm_config.yaml` for a team proxy:

```yaml
model_list:
  - model_name: claude-sonnet-4-6
    litellm_params:
      model: anthropic/claude-sonnet-4-6
      api_key: os.environ/ANTHROPIC_API_KEY
  - model_name: gpt-4o
    litellm_params:
      model: openai/gpt-4o
      api_key: os.environ/OPENAI_API_KEY

litellm_settings:
  success_callback: ["langfuse"]
  failure_callback: ["langfuse"]

general_settings:
  master_key: internal-token
  database_url: postgres://...
```

Start the proxy with Docker:

```bash
$ docker run -p 4000:4000 \
    -e ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY \
    -e OPENAI_API_KEY=$OPENAI_API_KEY \
    -v $(pwd)/litellm_config.yaml:/app/config.yaml \
    ghcr.io/berriai/litellm:main \
    --config /app/config.yaml
```

## On-Premise LLM (Air-Gapped)

In an environment without Internet access, you can run an LLM entirely locally via Ollama or vLLM.

### With Ollama (Local CPU/GPU)

```bash
# Download the model on a connected machine and export it
$ ollama pull llama3
$ ollama save llama3 llama3.tar

# Transfer llama3.tar to the internal network, then:
$ ollama load llama3.tar
$ ollama serve
```

```yaml
providers:
  air-gapped-llm:
    enabled: true
    baseUrl: http://localhost:11434/v1
    model: llama3
    maxTokens: 8000
    costPer1k: 0
    command: ""
```

### With vLLM (GPU Server)

```bash
$ pip install vllm
$ python -m vllm.entrypoints.openai.api_server \
    --model /models/mistral-7b-instruct \
    --host 0.0.0.0 \
    --port 8000
```

```yaml
providers:
  vllm-gpu:
    enabled: true
    baseUrl: http://gpu-server.internal:8000/v1
    model: mistral-7b-instruct
    maxTokens: 32000
    costPer1k: 0
    command: ""
```

> **Note**: The AIDEN prompt sanitizer (`sanitize: true`) remains active even with a local LLM. It protects against accidental secret injection in generated artifacts.

```bash
  azure: not configured
```

```bash
    --provider litellm
```

## Multi-Provider with API and CLI

You can mix CLI and API providers in the same configuration. The `auto` routing selects the best provider based on the task:

```yaml
# .aiden/aiden.config.yaml
provider: claude-code

routing:
  strategy: auto
  fallback: [litellm, local-llm]

providers:
  # CLI provider — analysis, review (native long context)
  claude-code:
    enabled: true
    command: claude
    model: claude-sonnet-4-6
    maxTokens: 100000
    costPer1k: 0.003

  # API provider — code generation via team proxy
  litellm:
    enabled: true
    baseUrl: http://localhost:4000/v1
    model: gpt-4o
    maxTokens: 128000
    costPer1k: 0.005
    command: ""

  # API provider — local fallback, zero cost
  local-llm:
    enabled: true
    baseUrl: http://localhost:11434/v1
    model: llama3
    maxTokens: 8000
    costPer1k: 0
    command: ""
```

In `auto` mode, the default routing becomes:

| Agent | Selected provider | Reason |
|---|---|---|
| Lead | claude-code | Request analysis |
| Analyst | claude-code | Complexity analysis |
| Planner | claude-code | Story decomposition |
| Dev | litellm | Code generation |
| QA | claude-code | Code review, long context |
| DocOps | claude-code | Documentation |

If `claude-code` is unavailable (CI without binary), AIDEN automatically falls back to `litellm` then `local-llm` according to the `fallback` list.

## Troubleshooting

### `fetch failed` — Connection Refused

The server is not started or the URL is incorrect.

```bash
# Verify the server is listening
$ curl http://localhost:4000/v1/models
# For Ollama: curl http://localhost:11434/v1/models
```

If `curl` also fails, the server is not started. Restart `litellm` or `ollama serve`.

### `401 Unauthorized`

The `apiKey` in the config doesn't match what the server expects.

- For LiteLLM: check the `master_key` value in `litellm_config.yaml`
- For direct OpenAI/Anthropic: verify the key is valid and not expired
- For an internal proxy: contact the platform team to renew the token

```bash
$ curl -H "Authorization: Bearer internal-token" \
       http://localhost:4000/v1/models
# Should return a JSON with the list of models
```

### `model not found` or `404 on /chat/completions`

The model name in AIDEN doesn't match what the server exposes.

```bash
# List available models on the server
$ curl http://localhost:4000/v1/models | python -m json.tool
# Look for the "id" field for each model
```

Adjust the `model` field in your AIDEN config to exactly match the returned identifier.

### High Latency on Long Workflows

API providers sometimes have higher latency than CLI providers, especially for local models on CPU. The default timeout is 120 seconds. For slow models:

```yaml
providers:
  local-llm:
    enabled: true
    baseUrl: http://localhost:11434/v1
    model: llama3:70b
    maxTokens: 8000
    costPer1k: 0
    timeout: 300000   # 5 minutes (in ms)
    command: ""
```

Also reduce `maxOutputCharsPerAgent` to limit the size of prompts sent:

```yaml
pipeline:
  maxOutputCharsPerAgent: 3000   # default: 8000
```

### Sanitizer Redacts Legitimate Data

The prompt sanitizer (`sanitize: true`) may incorrectly redact certain strings that resemble secrets. To add an exception or custom rule:

```yaml
sanitize: true
sanitizeRules:
  - pattern: "INTERNAL_REF_\\w+"
    replacement: "[INTERNAL_REF]"
```

To disable the sanitizer on a fully isolated environment (air-gapped, no sensitive data in prompts):

```yaml
sanitize: false
```

> **Warning**: Disabling the sanitizer in a cloud environment exposes your secrets if code with credentials is analyzed by AIDEN.
