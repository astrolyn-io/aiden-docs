# Installation

**Still writing specs by hand, debugging alone at 11 PM, and watching your backlog grow endlessly?** AIDEN changes the game — but first, you need to install it. Good news: it takes just 3 commands.

> **TL;DR** — For the impatient, here's the express version:
> ```bash
> pnpm add -g @astrolyn/aiden   # 1. Install
> aiden init                     # 2. Initialize in your project
> aiden flash "Hello world" --dry-run  # 3. Verify it works
> ```
> That's it. If you want to understand each step (and avoid the pitfalls), read on.

---

## Prerequisites

Before inviting AIDEN into your project, make sure you have the right environment. Think of it as gathering ingredients before the recipe:

- **Node.js 20+** — check with `node --version`
- **pnpm** — install with `npm install -g pnpm`
- At least one AI provider: Claude Code, Gemini CLI, OpenCode, or GitHub Copilot (or an API proxy like LiteLLM)

The provider is the "brain" that AIDEN will orchestrate. Without a provider, AIDEN is a conductor without musicians — it knows what to do, but there's nobody to play the score.

All set? Perfect, let's move on to the actual installation.

---

## Install AIDEN

One command, global installation:

```bash
$ pnpm add -g @astrolyn/aiden
```

Verify everything went smoothly:

```bash
$ aiden --version
0.7.0
```

> **Tip**: If you get `command not found`, add the pnpm global directory to your PATH: `export PATH="$(pnpm store path)/../../.pnpm:$PATH"`

The binary is there. Now you need to introduce AIDEN to your project — that's where `aiden init` comes in.

---

## Initialize a project

This is the step where AIDEN "gets acquainted" with your codebase. In your project directory:

```bash
$ aiden init
AIDEN initialized in .aiden/
  Config: .aiden/aiden.config.yaml
  Rules:  .aiden/rules/coding-standards.md
  Memory: .aiden/memory/
  Stack:  node, typescript, express
  Arch:   docker
  CI:     github-actions
  PM:     pnpm
```

What's happening under the hood? AIDEN automatically scans your project and detects everything it can. It's like a new developer reading your `package.json`, your `Dockerfile`, and your CI config on day one — except it does it in seconds:

- **Tech stack**: Node.js, TypeScript, React, Express, NestJS, Vue, Angular, Python, Go, Rust...
- **Architecture**: Docker, Docker Compose, Kubernetes
- **CI/CD**: GitHub Actions, GitLab CI, Jenkins, CircleCI
- **Package manager**: pnpm, yarn, npm, bun
- **Databases**: PostgreSQL, MySQL, Redis, MongoDB (from deps and docker-compose)
- **Monorepo**: Turborepo, Nx, Lerna, pnpm workspaces

This information is injected into every agent's context for more relevant results. The more AIDEN knows about your project, the more tailored its suggestions become.

Here's the structure created in your project:

```
.aiden/
  aiden.config.yaml    # Project configuration
  memory/            # Context, decisions, metrics
  rules/             # Enterprise rules
  templates/         # Custom templates
  features/          # Active features
```

The project is initialized — now let's make sure AIDEN can talk to your preferred AI.

---

## Configure the provider

By default, AIDEN uses Claude Code. If you prefer a different provider, a simple config change is all it takes:

```bash
$ aiden config set provider gemini
Set provider = gemini
```

Available providers: `claude-code`, `gemini`, `copilot`.

To verify your provider is accessible (because a tool you can't reach isn't much use):

```bash
$ aiden provider status
Provider status:
  claude-code: configured
  gemini: not found
  copilot: configured
```

If you see "configured" next to your provider, you're ready for the big moment: the first test.

---

## First test

Before unleashing AIDEN on a real feature, let's do a dry run. The `--dry-run` flag simulates execution without calling the provider — handy for verifying everything is properly wired:

```bash
$ aiden flash "Add hello.ts" --dry-run
[DRY RUN] Flash workflow: "Add hello.ts"
Mode: flash | Agents: lead, dev
```

If you see this output, congratulations: AIDEN is installed, initialized, and ready to work. The `--dry-run` shows what would be done without calling the provider.

## Slash mode (Claude Code)

If you already use Claude Code daily, there's an even more direct shortcut: slash commands. Zero extra installation, zero config — just commands in your Claude session:

- `/aiden-init` — initialize AIDEN
- `/aiden-flash` — fast mode
- `/aiden-standard` — full mode
- `/aiden-enterprise` — enterprise mode

Both modes (CLI and slash) write to the same `.aiden/` files. You can freely switch between the two without losing your context.

## Troubleshooting

Things don't always go as planned. Here are the classic errors and how to fix them in 30 seconds:

**`command not found: aiden`** — Make sure the pnpm global directory is in your PATH.

**`AIDEN not initialized`** — Run `aiden init` in your project directory.

**`No provider available`** — Install Claude Code (`npm install -g @anthropic-ai/claude-code`) or another provider.

**Provider timeout** — The default timeout is 120s. If your LLM is slow, increase it in `.aiden/aiden.config.yaml`.
