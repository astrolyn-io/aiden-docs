# Skills

> Une regle de code review, c'est bien. Un package complet avec instructions, regles, hooks et config qui s'active en une commande ? C'est un skill.

**Ce que vous allez apprendre :**
- Ce qu'est un skill AIDEN et pourquoi l'utiliser
- Quand choisir un skill plutot qu'une simple regle
- Comment creer, gerer et partager vos propres skills
- Le format `skill.yaml` et toutes ses options

## Qu'est-ce qu'un skill ?

Un **skill** est un package declaratif (YAML + Markdown) qui regroupe des instructions, regles et hooks injectables dans le pipeline AIDEN. C'est le mecanisme d'extensibilite principal pour les entreprises.

## Quand utiliser un skill vs une simple regle ?

Avant de creer un skill, posez-vous la question : une simple regle dans `.aiden/rules/` suffit-elle ?

| | **Regle simple** (`.aiden/rules/`) | **Skill** (`.aiden/skills/`) |
|---|---|---|
| **Cas d'usage** | Convention unique et transversale | Preoccupation complete (compliance, integration, workflow) |
| **Ciblage agents** | Envoyee a tous les agents | Cible des agents specifiques |
| **Filtrage par mode** | Toujours active | Activable par mode (flash, standard, enterprise) |
| **Hooks** | Non | Oui (on_feature_created, on_phase_end, etc.) |
| **Configuration** | Non | Oui (variables custom, env vars) |
| **Versioning** | Fichier seul | Package versionne, activable/desactivable |
| **Exemple** | "Pas de `any` en TypeScript" | "Compliance RGPD : instructions + regles + hooks de notification" |

**En resume** : si votre besoin tient en un fichier Markdown sans ciblage particulier, une regle suffit. Des que vous avez besoin de cibler des agents, filtrer par mode, ou ajouter des hooks, c'est un skill qu'il vous faut.

## Structure d'un skill

```
.aiden/skills/
  mon-skill/
    skill.yaml            # Manifeste : metadata, agents cibles, hooks
    instructions.md       # Instructions injectees dans les prompts agents
    rules/                # Regles additionnelles (optionnel)
      security.md
    hooks/                # Scripts hooks (optionnel)
      on_feature_created.sh
```

### `skill.yaml` -- Le manifeste

```yaml
name: mon-skill               # (requis) Identifiant unique
version: 1.0.0                # Version du skill
description: "Description"    # Description lisible
author: equipe-platform        # Auteur (optionnel)

# Agents qui recevront les instructions de ce skill
agents:
  - dev
  - qa

# Modes ou ce skill s'active (tous si omis)
modes:
  - standard
  - enterprise

# Hooks declares par ce skill
hooks:
  on_feature_created:
    - command: ./hooks/notify.sh
      onFailure: warn
  on_feature_completed:
    - command: ./hooks/close-ticket.sh
      onFailure: ignore

# Configuration custom (accessible dans les hooks via env vars)
config:
  jira_url: https://mycompany.atlassian.net
  project_key: PROJ

# Priorite d'injection (plus haut = injecte en premier)
priority: 10

# Actif ou non
enabled: true
```

### `instructions.md` -- Les instructions

Le contenu de ce fichier est injecte dans le prompt de chaque agent cible sous la section `--- Skills Instructions ---`.

```markdown
# Mon Skill

## For Dev Agent
- Toujours utiliser des types stricts
- Pas de `any` autorise

## For QA Agent
- Verifier la couverture de tests >= 80%
```

### `rules/` -- Regles additionnelles

Les fichiers `.md` dans ce dossier sont merges avec les regles du projet (`.aiden/rules/`) et injectes dans la section `--- Enterprise Rules ---` du prompt.

## Schema complet `skill.yaml`

| Cle | Type | Defaut | Description |
|---|---|---|---|
| `name` | string | (requis) | Identifiant unique du skill |
| `version` | string | `1.0.0` | Version semver |
| `description` | string | `""` | Description lisible |
| `author` | string | -- | Auteur ou equipe |
| `agents` | AgentRole[] | tous les agents | Agents qui recevront les instructions |
| `modes` | CTAMode[] | tous les modes | Modes ou le skill est actif |
| `hooks` | Record | -- | Hooks par evenement |
| `config` | Record | -- | Configuration custom |
| `priority` | number | `0` | Priorite d'injection (decroissant) |
| `enabled` | boolean | `true` | Activer/desactiver sans supprimer |
| `compatibility` | object | -- | Contraintes de version (voir ci-dessous) |

### Champ `compatibility`

```yaml
compatibility:
  aiden: ">=0.9.0"              # Version AIDEN minimum (range semver)
  skills:                        # Dependances entre skills
    secret-scan: ">=1.0.0"
  conflicts:                     # Skills incompatibles
    - legacy-security-check
```

### Valeurs `agents` possibles

`lead`, `analyst`, `planner`, `dev`, `qa`, `docops`

### Valeurs `modes` possibles

`flash`, `standard`, `enterprise`

### Evenements hooks

`on_feature_created`, `on_phase_start`, `on_phase_end`, `on_feature_completed`, `on_feature_blocked`, `on_agent_start`, `on_agent_output`, `on_agent_error`, `on_quality_gate_pass`, `on_quality_gate_fail`

## Marketplace de skills

AIDEN fournit un catalogue de skills officiels sur etagere. Installez uniquement ce dont vous avez besoin, quand vous en avez besoin.

### Rechercher des skills

```bash
$ aiden skills search security
$ aiden skills search --category compliance
$ aiden skills search --agent qa
```

### Installer un skill

```bash
$ aiden skills install owasp-check           # Depuis le registry (derniere version)
$ aiden skills install owasp-check@2.1.0     # Version specifique
$ aiden skills install ./mon-skill-local     # Depuis un chemin local
$ aiden skills install owasp-check --force   # Ecraser un skill existant
```

### Verifier les mises a jour

```bash
$ aiden skills outdated                      # Lister les skills avec version plus recente
$ aiden skills info owasp-check              # Details d'un skill
```

### Diagnostic de compatibilite

```bash
$ aiden skills doctor
  ✓ code-review-strict@1.1.0  — compatible
  ⚠ hipaa-guard@1.0.0          — necessite AIDEN >=1.0.0 (actuel: 0.8.0)
  ✗ legacy-check@0.3.0         — conflit avec owasp-check
```

### Configuration

```yaml
# aiden.config.yaml
skills:
  registryUrl: https://registry.aiden.dev/skills.json  # Registry custom
  autoUpdate: false
  trustedAuthors: [aiden-team]
```

## Skill Packs (bundles)

## Comment les skills sont charges

1. Au demarrage d'un workflow, l'orchestrateur charge tous les skills depuis `.aiden/skills/`
2. Les skills avec `enabled: false` sont filtres
3. Les skills sont tries par `priority` (plus haut = injecte en premier)
4. Pour chaque agent, seuls les skills qui le ciblent (et correspondent au mode) sont injectes
5. Les instructions apparaissent dans le prompt sous `--- Skills Instructions ---`
6. Les rules des skills sont mergees avec celles du projet sous `--- Enterprise Rules ---`

## Partager des skills entre equipes

### Git submodule

```bash
$ git submodule add https://github.com/org/aiden-skills-common .aiden/skills/common
```

### Copie directe

```bash
$ aiden skills add /path/to/shared-skills/compliance-rgpd
```

### Monorepo

Dans un monorepo, placez les skills partages a la racine et creez des liens symboliques :

```bash
# Racine du monorepo
shared-skills/
  compliance-rgpd/

# Dans chaque package
.aiden/skills/compliance-rgpd -> ../../../shared-skills/compliance-rgpd
```

## Bonnes pratiques

1. **Un skill = une preoccupation** : ne melangez pas securite et Jira dans le meme skill
2. **Instructions courtes et actionnables** : les LLMs preferent des listes de regles claires
3. **Ciblez les agents pertinents** : ne ciblez pas tous les agents si seul le Dev est concerne
4. **Utilisez les modes** : les skills de conformite n'ont pas besoin d'etre actifs en mode flash
5. **Priorite** : les skills de securite/conformite devraient avoir une priorite haute (>= 20)
6. **Committez vos skills** : `.aiden/skills/` doit etre versionne avec le code
