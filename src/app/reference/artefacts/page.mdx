# Reference Artefacts

Chaque agent laisse une trace. Voici les 7 types d'artefacts que vous trouverez dans .aiden/features/ — et ce qu'ils contiennent.

---

## Format commun

Chaque artefact est un fichier Markdown avec frontmatter YAML. Ce format uniforme permet le suivi, le versionnage et l'injection automatique dans les prompts :

```yaml
---
type: spec
feature: export-csv
version: 1
created: 2026-03-26
author: planner
status: draft
provider: claude-code
tokensUsed: 3421
mode: standard
---
```

Statuts possibles : `draft`, `review`, `approved`, `implemented`.

---

## Les 7 artefacts

### brief.md

Genere par l'**Analyst**. Resume la demande, contexte, objectif, contraintes et risques. C'est le point de depart de toute analyse — les agents suivants s'appuient dessus.

### spec.md

Genere par le **Planner**. Specification allegee (1-2 pages) avec exigences et criteres d'acceptation. Sert de contrat entre la planification et le developpement.

### story.md (dans stories/)

Genere par le **Planner**. User stories numerotees (001-*, 002-*) avec criteres d'acceptation. Chaque story est une unite de travail independante pour l'agent Dev.

### review.md

Genere par le **QA**. Resultat du code review avec score, issues et suggestions. Si le score est insuffisant, le workflow relance automatiquement le Dev avec le feedback.

### change-summary.md

Genere par le **Dev** (mode flash). Resume des changements effectues — fichiers modifies, tests generes, decisions prises.

### architecture-decision.md

Genere par le **Planner** (mode enterprise). ADR : contexte, decision, alternatives, consequences. Trace les choix architecturaux pour reference future.

### quality-report.md

Genere par le **QA** (mode enterprise). Rapport complet avec metriques, securite et recommandations. Inclut la verification explicite de chaque critere d'acceptation.

---

## Emplacement

Tous les artefacts sont dans `.aiden/features/<feature-slug>/`.
