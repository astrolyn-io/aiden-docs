# AIDEN en équipe

> **Ce que vous allez apprendre**
> - Configurer AIDEN pour une équipe de 5 à 50+ développeurs
> - Partager les règles, conventions et templates
> - Workflow Git avec AIDEN
> - Onboarding d'un nouveau membre en 30 minutes

## Configuration équipe

### Règles partagées dans Git

Commitez `.aiden/rules/` dans votre repo. Toute l'équipe utilise les mêmes standards automatiquement.

```
.aiden/
  rules/
    coding-standards.md         # Standards de code communs
    security-policy.md          # Politique de sécurité
    architecture-principles.md  # Principes d'architecture
    api-guidelines.md           # Standards API REST
```

Chaque règle est un fichier Markdown injecté automatiquement dans les prompts des agents.

### Config globale (multi-projets)

Pour les règles qui s'appliquent à **tous les projets** de l'organisation, utilisez la config globale :

```yaml
# ~/.aiden/enterprise.config.yaml (sur chaque poste)
company: MonEntreprise
defaults:
  mode: standard
  lang: fr
  provider: claude-code

security:
  no_secrets_in_artefacts: true
  scan_dependencies: true
  require_security_review: true

compliance:
  enabled: true
  frameworks: [RGPD, ISO27001]
  audit_log: true

coding:
  test_coverage_min: 80
  max_pr_size: 400
  commit_convention: conventional
```

Distribuez ce fichier via le script de provisioning des postes développeurs.

### Profiles par équipe

Si votre organisation a plusieurs équipes avec des stacks différentes :

```yaml
# .aiden/aiden.config.yaml
profiles:
  backend:
    rules: [coding-standards.md, api-guidelines.md, database-rules.md]
    provider: claude-code
    mode: standard

  frontend:
    rules: [coding-standards-front.md, accessibility.md, ui-guidelines.md]
    provider: gemini
    mode: standard

  data:
    rules: [coding-standards-python.md, data-pipeline-rules.md]
    provider: gemini
    mode: standard
```

## Workflow Git avec AIDEN

### Workflow recommandé

```
1. Créer une branche       git checkout -b feature/export-csv
2. Lancer AIDEN              aiden standard "Ajouter export CSV des rapports"
3. AIDEN génère              code + tests + artefacts dans .aiden/features/
4. Review humaine          Le reviewer lit review.md + spec.md + le diff
5. Commiter                git add -A && git commit -m "feat: export CSV"
6. Push + PR               Les artefacts aident le reviewer
7. Merge + clean           La feature reste archivée dans .aiden/features/
```

### Convention de branches

| Type de branche | Mode AIDEN recommandé |
|----------------|---------------------|
| `fix/<slug>` | `aiden flash` |
| `feature/<slug>` | `aiden standard` |
| `refactor/<slug>` | `aiden standard` |
| `release/<slug>` | `aiden enterprise` |
| `hotfix/<slug>` | `aiden flash` |

### Que commiter dans Git ?

| Répertoire | Commiter ? | Raison |
|-----------|------------|--------|
| `.aiden/aiden.config.yaml` | **Oui** | Configuration partagée |
| `.aiden/rules/` | **Oui** | Standards d'équipe |
| `.aiden/templates/` | **Oui** | Templates personnalisés |
| `.aiden/features/` | **Oui** | Artefacts = documentation vivante |
| `.aiden/memory/context.yaml` | **Optionnel** | État courant (utile pour la reprise) |
| `.aiden/memory/decisions.jsonl` | **Oui** | Audit trail |
| `.aiden/memory/metrics.jsonl` | **Optionnel** | Métriques (utile pour le suivi) |

### .gitignore recommandé

```gitignore
# AIDEN — garder les artefacts, ignorer les fichiers temporaires
.aiden/memory/*.lock
```

## Politique de review avec AIDEN

### Review basée sur les artefacts

Quand un développeur ouvre une PR après un `aiden standard`, le reviewer dispose de :

| Artefact | Ce que le reviewer vérifie |
|----------|---------------------------|
| `brief.md` | L'analyse est-elle correcte ? Les risques identifiés ? |
| `spec.md` | La spec correspond-elle au besoin ? |
| `stories/*.md` | Les critères d'acceptation sont-ils complets ? |
| `review.md` | Le QA a-t-il identifié des problèmes ? Tous les critères PASS ? |
| `decisions.jsonl` | Les décisions IA sont-elles raisonnables ? |

Le reviewer humain se concentre sur la **cohérence métier** et les **choix d'architecture**, pas sur le style ou les tests (déjà vérifiés par le QA).

### Niveaux de review par mode

| Mode | Review requise |
|------|---------------|
| Flash | Self-review du dev (le QA est skip) |
| Standard | review.md du QA + review humaine sur la PR |
| Enterprise | review.md + review humaine + approbation tech lead |

## Onboarding d'un nouveau membre

### Checklist (30 minutes)

```
5 min   Installer Node 20+ et pnpm
2 min   pnpm add -g @aiden/cli
1 min   Copier ~/.aiden/enterprise.config.yaml (via provisioning)
2 min   Cloner le repo (les règles sont incluses dans .aiden/rules/)
1 min   aiden provider status  — vérifier le provider
5 min   aiden flash "Ajouter un log info au démarrage"  — premier flash
5 min   aiden status + explorer les artefacts générés
10 min  Lire les artefacts d'une feature existante dans .aiden/features/
```

### Valeur onboarding des artefacts

Les artefacts AIDEN dans `.aiden/features/` sont des **documents d'onboarding vivants**. Un nouveau membre peut lire :

- Le `brief.md` pour comprendre **pourquoi** une feature existe
- Le `spec.md` pour comprendre **comment** elle a été conçue
- Le `review.md` pour comprendre **quels problèmes** ont été identifiés
- Le `decisions.jsonl` pour comprendre **quelles décisions IA** ont été prises

C'est plus rapide que lire le code et deviner l'intention.

### Capitaliser avec les retex

Les **retex** (retours d'expérience) permettent à l'équipe de capitaliser les leçons apprises :

```bash
# Apres une feature reussie
$ aiden retex add --title "Migration ORM incrementale" \
  --context prisma,drizzle,migration \
  --outcome success \
  --lesson "Migrer table par table est plus sur qu'un big bang" \
  --recommendation "Garder les deux ORMs en parallele pendant la migration"

# Apres un echec
$ aiden retex add --title "Cache Redis sans TTL" \
  --context redis,cache,bug \
  --outcome failure \
  --lesson "Un cache sans TTL a cause un OOM en production" \
  --recommendation "Toujours definir un TTL sur les cles Redis"

# Consulter les retex pertinents
$ aiden retex search migration
```

Les retex pertinents sont automatiquement injectés dans les prompts des agents. L'équipe apprend collectivement de ses expériences.

## Métriques d'équipe

### Suivi d'adoption

```bash
# Métriques globales
$ aiden metrics

# Export CSV pour dashboard
$ aiden metrics --export csv > metriques-equipe-mars.csv
```

### KPI équipe

| Métrique | Cible | Comment |
|----------|-------|---------|
| % devs actifs sur AIDEN | > 70% | metrics.jsonl par agent unique |
| Features standard vs flash | > 30% standard | decisions.jsonl type=mode_selection |
| Coût moyen par dev/mois | < $80 | metrics.jsonl agrégé |
| Taux circuit breaker | < 10% | decisions.jsonl type=circuit_breaker |

## Anti-patterns à éviter

| Anti-pattern | Pourquoi c'est un problème | Solution |
|-------------|---------------------------|----------|
| Forcer AIDEN sur toute l'équipe d'un coup | Rejet et frustration | Adopter progressivement : pilote → équipes → général |
| Utiliser enterprise pour tout | Sur-ingénierie, lenteur | Flash pour les fix, standard pour les features, enterprise pour les projets critiques |
| Ne pas commiter les artefacts | Perte de traçabilité et d'onboarding | Toujours commiter `.aiden/features/` avec le code |
| Ignorer le review.md | Perdre la valeur du QA automatique | Le reviewer humain doit lire le review.md en premier |
| Des règles trop strictes au début | Friction, contournement | Commencer avec des règles légères, durcir progressivement |
