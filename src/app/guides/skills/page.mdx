## Qu'est-ce qu'un skill ?

Un **skill** est un package declaratif (YAML + Markdown) qui regroupe des instructions, regles et hooks injectables dans le pipeline CTA. C'est le mecanisme d'extensibilite principal pour les entreprises.

Contrairement aux regles simples (`.cta/rules/`), un skill :
- **Cible des agents specifiques** (ex: uniquement Dev et QA)
- **Filtre par mode** (ex: uniquement en enterprise)
- **Declare des hooks** executables a chaque etape du workflow
- **Porte sa propre configuration** accessible dans les hooks
- **Est versionne et activable/desactivable**

## Structure d'un skill

```
.cta/skills/
  mon-skill/
    skill.yaml            # Manifeste : metadata, agents cibles, hooks
    instructions.md       # Instructions injectees dans les prompts agents
    rules/                # Regles additionnelles (optionnel)
      security.md
    hooks/                # Scripts hooks (optionnel)
      on_feature_created.sh
```

### `skill.yaml` — Le manifeste

```yaml
name: mon-skill               # (requis) Identifiant unique
version: 1.0.0                # Version du skill
description: "Description"    # Description lisible
author: equipe-platform        # Auteur (optionnel)

# Agents qui recevront les instructions de ce skill
agents:
  - dev
  - qa

# Modes ou ce skill s'active (tous si omis)
modes:
  - standard
  - enterprise

# Hooks declares par ce skill
hooks:
  on_feature_created:
    - command: ./hooks/notify.sh
      onFailure: warn
  on_feature_completed:
    - command: ./hooks/close-ticket.sh
      onFailure: ignore

# Configuration custom (accessible dans les hooks via env vars)
config:
  jira_url: https://mycompany.atlassian.net
  project_key: PROJ

# Priorite d'injection (plus haut = injecte en premier)
priority: 10

# Actif ou non
enabled: true
```

### `instructions.md` — Les instructions

Le contenu de ce fichier est injecte dans le prompt de chaque agent cible sous la section `--- Skills Instructions ---`.

```markdown
# Mon Skill

## For Dev Agent
- Toujours utiliser des types stricts
- Pas de `any` autorise

## For QA Agent
- Verifier la couverture de tests >= 80%
```

### `rules/` — Regles additionnelles

Les fichiers `.md` dans ce dossier sont merges avec les regles du projet (`.cta/rules/`) et injectes dans la section `--- Enterprise Rules ---` du prompt.

## Schema complet `skill.yaml`

| Cle | Type | Defaut | Description |
|---|---|---|---|
| `name` | string | (requis) | Identifiant unique du skill |
| `version` | string | `1.0.0` | Version semver |
| `description` | string | `""` | Description lisible |
| `author` | string | — | Auteur ou equipe |
| `agents` | AgentRole[] | tous les agents | Agents qui recevront les instructions |
| `modes` | CTAMode[] | tous les modes | Modes ou le skill est actif |
| `hooks` | Record | — | Hooks par evenement |
| `config` | Record | — | Configuration custom |
| `priority` | number | `0` | Priorite d'injection (decroissant) |
| `enabled` | boolean | `true` | Activer/desactiver sans supprimer |

### Valeurs `agents` possibles

`lead`, `analyst`, `planner`, `dev`, `qa`, `docops`

### Valeurs `modes` possibles

`flash`, `standard`, `enterprise`

### Evenements hooks

`on_feature_created`, `on_phase_start`, `on_phase_end`, `on_feature_completed`, `on_feature_blocked`, `on_agent_start`, `on_agent_output`, `on_agent_error`, `on_quality_gate_pass`, `on_quality_gate_fail`

## Commandes CLI

### Creer un nouveau skill

```bash
cta skills init mon-skill
```

Cree `.cta/skills/mon-skill/` avec un `skill.yaml` template et un `instructions.md` vide.

### Lister les skills installes

```bash
cta skills list
```

Affiche nom, version, agents cibles et statut (enabled/disabled).

### Installer un skill externe

```bash
cta skills add /chemin/vers/skill
```

Copie le dossier skill dans `.cta/skills/`.

### Supprimer un skill

```bash
cta skills remove mon-skill
```

### Activer/Desactiver

```bash
cta skills enable mon-skill
cta skills disable mon-skill
```

Modifie `enabled` dans `skill.yaml` sans supprimer le skill.

## Exemples de skills

CTA fournit 3 templates de skills dans `templates/skills/` :

### `code-review-strict`

Enforce des standards de code review stricts pour Dev et QA : types explicites, pas de `any`, JSDoc obligatoire, couverture de tests >= 80%.

```bash
cp -r templates/skills/code-review-strict .cta/skills/
```

### `compliance-rgpd`

Checks de conformite RGPD pour le code manipulant des donnees personnelles. Cible Analyst, Dev et QA en modes standard et enterprise.

### `jira-sync`

Synchronise les features CTA avec Jira. Declare des hooks sur `on_feature_created` et `on_feature_completed`. Desactive par defaut (a configurer avec votre URL Jira).

## Comment les skills sont charges

1. Au demarrage d'un workflow, l'orchestrateur charge tous les skills depuis `.cta/skills/`
2. Les skills avec `enabled: false` sont filtres
3. Les skills sont tries par `priority` (plus haut = injecte en premier)
4. Pour chaque agent, seuls les skills qui le ciblent (et correspondent au mode) sont injectes
5. Les instructions apparaissent dans le prompt sous `--- Skills Instructions ---`
6. Les rules des skills sont mergees avec celles du projet sous `--- Enterprise Rules ---`

## Partager des skills entre equipes

### Git submodule

```bash
git submodule add https://github.com/org/cta-skills-common .cta/skills/common
```

### Copie directe

```bash
cta skills add /path/to/shared-skills/compliance-rgpd
```

### Monorepo

Dans un monorepo, placez les skills partages a la racine et creez des liens symboliques :

```bash
# Racine du monorepo
shared-skills/
  compliance-rgpd/

# Dans chaque package
.cta/skills/compliance-rgpd -> ../../../shared-skills/compliance-rgpd
```

## Bonnes pratiques

1. **Un skill = une preoccupation** : ne melangez pas securite et Jira dans le meme skill
2. **Instructions courtes et actionnables** : les LLMs preferent des listes de regles claires
3. **Ciblez les agents pertinents** : ne ciblez pas tous les agents si seul le Dev est concerne
4. **Utilisez les modes** : les skills de conformite n'ont pas besoin d'etre actifs en mode flash
5. **Priorite** : les skills de securite/conformite devraient avoir une priorite haute (>= 20)
6. **Committez vos skills** : `.cta/skills/` doit etre versionne avec le code
