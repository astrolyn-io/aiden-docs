## Qu'est-ce qu'un skill ?

Un **skill** est un package déclaratif (YAML + Markdown) qui regroupe des instructions, règles et hooks injectables dans le pipeline AIDEN. C'est le mécanisme d'extensibilité principal pour les entreprises.

Contrairement aux règles simples (`.aiden/rules/`), un skill :
- **Cible des agents spécifiques** (ex: uniquement Dev et QA)
- **Filtre par mode** (ex: uniquement en enterprise)
- **Déclare des hooks** exécutables à chaque étape du workflow
- **Porte sa propre configuration** accessible dans les hooks
- **Est versionné et activable/désactivable**

## Structure d'un skill

```
.aiden/skills/
  mon-skill/
    skill.yaml            # Manifeste : metadata, agents cibles, hooks
    instructions.md       # Instructions injectees dans les prompts agents
    rules/                # Regles additionnelles (optionnel)
      security.md
    hooks/                # Scripts hooks (optionnel)
      on_feature_created.sh
```

### `skill.yaml` — Le manifeste

```yaml
name: mon-skill               # (requis) Identifiant unique
version: 1.0.0                # Version du skill
description: "Description"    # Description lisible
author: equipe-platform        # Auteur (optionnel)

# Agents qui recevront les instructions de ce skill
agents:
  - dev
  - qa

# Modes ou ce skill s'active (tous si omis)
modes:
  - standard
  - enterprise

# Hooks declares par ce skill
hooks:
  on_feature_created:
    - command: ./hooks/notify.sh
      onFailure: warn
  on_feature_completed:
    - command: ./hooks/close-ticket.sh
      onFailure: ignore

# Configuration custom (accessible dans les hooks via env vars)
config:
  jira_url: https://mycompany.atlassian.net
  project_key: PROJ

# Priorite d'injection (plus haut = injecte en premier)
priority: 10

# Actif ou non
enabled: true
```

### `instructions.md` — Les instructions

Le contenu de ce fichier est injecté dans le prompt de chaque agent cible sous la section `--- Skills Instructions ---`.

```markdown
# Mon Skill

## For Dev Agent
- Toujours utiliser des types stricts
- Pas de `any` autorise

## For QA Agent
- Verifier la couverture de tests >= 80%
```

### `rules/` — Règles additionnelles

Les fichiers `.md` dans ce dossier sont mergés avec les règles du projet (`.aiden/rules/`) et injectés dans la section `--- Enterprise Rules ---` du prompt.

## Schéma complet `skill.yaml`

| Clé | Type | Défaut | Description |
|---|---|---|---|
| `name` | string | (requis) | Identifiant unique du skill |
| `version` | string | `1.0.0` | Version semver |
| `description` | string | `""` | Description lisible |
| `author` | string | — | Auteur ou équipe |
| `agents` | AgentRole[] | tous les agents | Agents qui recevront les instructions |
| `modes` | CTAMode[] | tous les modes | Modes où le skill est actif |
| `hooks` | Record | — | Hooks par événement |
| `config` | Record | — | Configuration custom |
| `priority` | number | `0` | Priorité d'injection (décroissant) |
| `enabled` | boolean | `true` | Activer/désactiver sans supprimer |

### Valeurs `agents` possibles

`lead`, `analyst`, `planner`, `dev`, `qa`, `docops`

### Valeurs `modes` possibles

`flash`, `standard`, `enterprise`

### Événements hooks

`on_feature_created`, `on_phase_start`, `on_phase_end`, `on_feature_completed`, `on_feature_blocked`, `on_agent_start`, `on_agent_output`, `on_agent_error`, `on_quality_gate_pass`, `on_quality_gate_fail`

## Commandes CLI

### Créer un nouveau skill

```bash
$ aiden skills init mon-skill
```

Crée `.aiden/skills/mon-skill/` avec un `skill.yaml` template et un `instructions.md` vide.

### Lister les skills installés

```bash
$ aiden skills list
```

Affiche nom, version, agents cibles et statut (enabled/disabled).

### Installer un skill externe

```bash
$ aiden skills add /chemin/vers/skill
```

Copie le dossier skill dans `.aiden/skills/`.

### Supprimer un skill

```bash
$ aiden skills remove mon-skill
```

### Activer/Désactiver

```bash
$ aiden skills enable mon-skill
$ aiden skills disable mon-skill
```

Modifie `enabled` dans `skill.yaml` sans supprimer le skill.

## Exemples de skills

AIDEN fournit 3 templates de skills dans `templates/skills/` :

### `code-review-strict`

Enforce des standards de code review stricts pour Dev et QA : types explicites, pas de `any`, JSDoc obligatoire, couverture de tests >= 80%.

```bash
$ cp -r templates/skills/code-review-strict .aiden/skills/
```

### `compliance-rgpd`

Checks de conformité RGPD pour le code manipulant des données personnelles. Cible Analyst, Dev et QA en modes standard et enterprise.

### `jira-sync`

Synchronise les features AIDEN avec Jira. Déclare des hooks sur `on_feature_created` et `on_feature_completed`. Désactivé par défaut (à configurer avec votre URL Jira).

## Comment les skills sont chargés

1. Au démarrage d'un workflow, l'orchestrateur charge tous les skills depuis `.aiden/skills/`
2. Les skills avec `enabled: false` sont filtrés
3. Les skills sont triés par `priority` (plus haut = injecté en premier)
4. Pour chaque agent, seuls les skills qui le ciblent (et correspondent au mode) sont injectés
5. Les instructions apparaissent dans le prompt sous `--- Skills Instructions ---`
6. Les rules des skills sont mergées avec celles du projet sous `--- Enterprise Rules ---`

## Partager des skills entre équipes

### Git submodule

```bash
$ git submodule add https://github.com/org/aiden-skills-common .aiden/skills/common
```

### Copie directe

```bash
$ aiden skills add /path/to/shared-skills/compliance-rgpd
```

### Monorepo

Dans un monorepo, placez les skills partagés à la racine et créez des liens symboliques :

```bash
# Racine du monorepo
shared-skills/
  compliance-rgpd/

# Dans chaque package
.aiden/skills/compliance-rgpd -> ../../../shared-skills/compliance-rgpd
```

## Bonnes pratiques

1. **Un skill = une préoccupation** : ne mélangez pas sécurité et Jira dans le même skill
2. **Instructions courtes et actionnables** : les LLMs préfèrent des listes de règles claires
3. **Ciblez les agents pertinents** : ne ciblez pas tous les agents si seul le Dev est concerné
4. **Utilisez les modes** : les skills de conformité n'ont pas besoin d'être actifs en mode flash
5. **Priorité** : les skills de sécurité/conformité devraient avoir une priorité haute (>= 20)
6. **Committez vos skills** : `.aiden/skills/` doit être versionné avec le code
