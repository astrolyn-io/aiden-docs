## Les 6 agents

### Lead (Orchestrateur)

Toujours actif. Analyse la demande, choisit le mode, orchestre les agents, applique les quality gates et circuit breakers.

### Analyst

Actif en standard et enterprise. Scanne le code, pose 3-5 questions, crée le brief, estime la complexité.

### Planner

Actif en standard et enterprise. Crée la spec allégée (1-2 pages), décompose en stories, écrit les ADR si nécessaire.

### Dev

Toujours actif. Implémente le code, écrit les tests, fait un self-review. Guardrails : rappel objectif toutes les 5 étapes, max 10 fichiers, circuit breaker.

### QA

Actif en standard et enterprise. Review du code, vérification des critères d'acceptation, scan sécurité OWASP basique, rapport qualité.

### DocOps

Actif en enterprise uniquement. Génère la documentation, changelog, vérifie la conformité, crée les runbooks.

## Comment les agents communiquent (Pipeline)

Les agents ne travaillent pas en isolation. Chaque agent reçoit un `PipelineContext` contenant :
- La **feature request** originale
- Les **outputs de tous les agents précédents** (tronqués à 8000 chars par agent, configurable)
- Le **contexte projet** réel (tech stack, architecture, CI — détectés par `aiden init`)

Flux concret en mode standard :

```
Analyst execute() → output.content + output.structured
    ↓
Planner recoit [ANALYST]: dans son prompt → output.content + structured.stories
    ↓
Dev recoit [ANALYST]: + [PLANNER]: → implemente selon les stories
    ↓
QA recoit [ANALYST]: + [PLANNER]: + [DEV]: → review le code du Dev
```

### Outputs structurés

Chaque agent retourne des données typées dans `output.structured` :

| Agent | Champs structurés |
|-------|------------------|
| Analyst | `complexity` (1-5), `risks` (string[]), `questions` (string[]) |
| Planner | `stories` (array of id/title/description), `estimatedTokens` (number) |
| Dev | `filesChanged` (string[]), `testsGenerated` (string[]) |
| QA | `passed` (boolean), `score` (1-10), `issues` (array of severity/message), `criteriaResults` (array of story/criterion/passed/reason) |
| DocOps | `docsGenerated` (string[]) |

L'orchestrateur utilise ces champs pour les quality gates et le feedback loop.

### Vérification croisée QA ↔ Planner

Le QA ne fait pas une review générique. Il extrait les stories et critères d'acceptation de l'output du Planner (`structured.stories`) et les inclut explicitement dans sa review :

```
--- ACCEPTANCE CRITERIA TO VERIFY ---
Story: Setup Database
  - Tables are created
  - Migrations run successfully

For each criterion: PASS or FAIL with reason.
```

Le QA retourne `structured.criteriaResults` avec le résultat de chaque critère, permettant un suivi précis de la couverture fonctionnelle.

### Routage multi-provider par agent

Chaque agent reçoit le provider optimal pour sa tâche via le router :

| Agent | TaskType | Provider par défaut (auto) |
|-------|----------|---------------------------|
| Analyst | analysis | claude-code |
| Planner | planning | claude-code |
| Dev | code_gen | gemini |
| QA | code_review | claude-code |
| DocOps | documentation | claude-code |

L'orchestrateur appelle `router.route(taskType)` avant chaque exécution. Si le provider préféré est indisponible, le fallback prend le relais.

### Feedback loop QA → Dev

Si le QA échoue (`structured.passed = false`), l'orchestrateur :
1. Relance le Dev avec le feedback du QA dans `previousOutputs`
2. Relance le QA sur le nouveau code
3. Répète max 2 fois. Si toujours en échec, le workflow continue.

### Quality gates

Après la phase **discovery** : le gate vérifie que l'output de l'Analyst contient du contenu et une complexité détectée. Sinon, la feature est bloquée.

Après la phase **planning** : le gate vérifie que la spec contient du contenu et au moins une story. Sinon, blocage.

## Invocation manuelle

Vous pouvez invoquer un agent en dehors d'un workflow :

```bash
$ aiden agent analyst "Analyser le module billing"
Invoking agent "analyst" with: "Analyser le module billing"
```

```bash
$ aiden agent qa "Reviewer les changements du dernier commit"
```

> **Astuce** : L'invocation manuelle est utile pour des analyses ponctuelles sans créer de feature.
