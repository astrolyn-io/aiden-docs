## Providers supportes

AIDEN supporte 4 providers CLI nativement et permet de brancher n'importe quel provider API compatible OpenAI :

| Provider | Commande | Type | Forces |
|---|---|---|---|
| Claude Code | `claude` | CLI | Analyse, review, long contexte |
| Gemini CLI | `gemini` | CLI | Generation code, multi-fichier |
| OpenCode | `opencode` | CLI | Open source, multi-provider via config |
| GitHub Copilot | `gh copilot` | CLI | Completion, quick fix |
| Custom API | `fetch()` | HTTP | N'importe quel endpoint OpenAI-compatible |

## Configuration

Dans `.aiden/aiden.config.yaml` :

```yaml
provider: claude-code
providers:
  claude-code:
    enabled: true
    command: claude
    model: claude-sonnet-4-6
    maxTokens: 100000
    costPer1k: 0.003
  gemini:
    enabled: true
    command: gemini
    model: gemini-2.5-pro
    maxTokens: 1000000
    costPer1k: 0.001
  opencode:
    enabled: true
    command: opencode
    model: opencode
    maxTokens: 200000
    costPer1k: 0.002
  copilot:
    enabled: true
    command: gh
    model: copilot
    maxTokens: 50000
    costPer1k: 0
```

## Provider API custom (proxy LLM)

AIDEN peut se connecter a n'importe quel endpoint compatible OpenAI via l'adapter API. Cela permet de brancher un proxy LLM (LiteLLM, Cloudflare AI Gateway) ou une API directe.

```yaml
providers:
  litellm:
    enabled: true
    baseUrl: http://localhost:4000/v1
    model: claude-sonnet-4-6
    maxTokens: 100000
    costPer1k: 0.003
    command: ""
  openai-direct:
    enabled: true
    baseUrl: https://api.openai.com/v1
    apiKey: sk-...
    model: gpt-4o
    maxTokens: 128000
    costPer1k: 0.005
    command: ""
```

Quand `baseUrl` est defini, AIDEN utilise l'adapter HTTP (`fetch()`) au lieu du subprocess CLI. Cela fonctionne avec :
- **LiteLLM** : proxy open source unifiant 100+ APIs
- **Cloudflare AI Gateway** : gateway manage avec caching et analytics
- **API directes** : OpenAI, Anthropic (via proxy), Google AI
- **Modeles locaux** : Ollama, vLLM, tout endpoint OpenAI-compatible

## Routage

4 strategies disponibles :

- **prefer** (defaut) : utilise le provider prefere, fallback si indisponible
- **auto** : choisit le meilleur provider selon la tache
- **fixed** : un seul provider toujours
- **cost-optimized** : minimise les couts

```yaml
routing:
  strategy: auto
  fallback: [claude-code, gemini, opencode, copilot]
```

En mode `auto`, chaque agent recoit le provider optimal :

| Tache | Provider par defaut |
|---|---|
| Analyse, Planning, Documentation | claude-code |
| Generation code, Tests | gemini |
| Quick fix, PR review | copilot |

## Budget tokens

```yaml
budget:
  monthlyLimit: 500000
  alertAt: 80
  perFeature:
    flash: 5000
    standard: 20000
    enterprise: 100000
```

> **Information** : L'alerte se declenche a 80% du budget. AIDEN bloque les appels si le budget est depasse.

## Filtrage securite

AIDEN filtre automatiquement les donnees sensibles avant de les envoyer aux providers :

- Bearer tokens
- API keys (`sk-`, `ghp_`, `github_pat_`)
- Mots de passe
- Cles privees (PEM)
- Connection strings (postgres, mongodb, redis)
- AWS access keys

Le filtrage est actif par defaut. Pour le desactiver (usage local uniquement) :

```yaml
sanitize: false
```

## Provider factory dynamique

AIDEN instancie les providers automatiquement depuis la config. Pas besoin de modifier le code pour ajouter un provider :

- **Nom connu** (claude-code, gemini, copilot, opencode) → adapter CLI specifique
- **`baseUrl` defini** → adapter API HTTP (fetch)
- **Disabled** → ignore

## Communication avec les providers

**CLI adapters** envoient les prompts via **stdin pipe** (pas comme argument shell). Cela permet des prompts de taille illimitee.

**API adapters** utilisent `POST /chat/completions` avec le format OpenAI standard.

- **Timeout** : 120 secondes par defaut
- **Format review** : SCORE/PASSED/ISSUES avec fallback heuristique
- **Disponibilite** : CLI verifie `--version`, API verifie `GET /models`
