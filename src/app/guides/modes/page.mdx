# Modes

> **Ce que vous allez apprendre**
> - Les 3 modes : flash, standard, enterprise
> - Quand utiliser chaque mode
> - Le passage dynamique entre modes

## Comparaison

| Fonctionnalité | Flash | Standard | Enterprise |
|---|---|---|---|
| Temps | < 30 min | 2h - 1 jour | 1 jour - 1 semaine |
| Agents | Lead, Dev | +Analyst, Planner, QA | +DocOps |
| Brief | Non | Oui | Oui |
| Spec | Non | Oui | Oui + ADR |
| Review | Non | Oui | Oui + rapport qualité |
| Approbation humaine | Non | Non | Oui |

## Mode Flash

Pour les changements simples : bug fix, petite feature, refactoring ciblé.

```bash
$ aiden flash "Fix le bug #123"
```

Quand l'utiliser :
- Moins de 3 fichiers impactés
- Pas de nouvelle API
- Complexité faible (1-2/5)

## Mode Standard

Pour les features moyennes nécessitant planification.

```bash
$ aiden standard "Nouvelle API users"
```

Quand l'utiliser :
- 3 à 15 fichiers impactés
- Nouvelle API ou intégration
- Complexité moyenne (3-4/5)

## Mode Enterprise

Pour les projets critiques avec compliance.

```bash
$ aiden enterprise "Migration authentification"
```

Quand l'utiliser :
- Plus de 15 fichiers
- Nouvelle architecture
- Compliance requise (RGPD, sécurité)

### Approbation humaine

En mode enterprise, **chaque phase requiert une approbation humaine** avant de passer à la suivante. À la fin de chaque phase, le CLI affiche :

```
[APPROVAL] Phase "discovery" for "migration-auth" completed.
  Agent: analyst | Tokens: 2340
  Preview: COMPLEXITY: 4 RISKS: - Data loss during migration...
  Approve? (y/n):
```

Si vous répondez `n`, le workflow est avorté avec le statut `aborted`.

Pour bypasser les approbations (par exemple en CI/CD) :

```bash
$ aiden enterprise "Migration auth" --auto-approve
```

> **Attention** : `--auto-approve` désactive toute validation humaine. À utiliser uniquement dans les pipelines automatisés.

## Détection automatique du mode

Le Lead analyse la description de la feature avec des heuristiques pour suggérer le mode optimal :

- `"Fix typo in README"` → **flash** (faible complexité, 1 fichier)
- `"Add new REST API with OAuth"` → **standard** (nouvelle API, dépendance externe)
- `"Migrate monolith to microservices with RGPD compliance"` → **enterprise** (nouvelle architecture, compliance)

La détection prend aussi en compte le contexte projet (nombre de technologies, architecture).

## Passage dynamique avec re-routing

AIDEN peut changer de mode en cours de route **sans perdre le travail** et en exécutant les phases manquantes :

```bash
# Démarre en flash, bascule en standard si complexité détectée
$ aiden flash "Export CSV"
AIDEN Lead > Complexité détectée (5 fichiers). Passer en standard ? [Y/n]
```

Quand vous passez de flash à standard, `switchMode()` :
1. Identifie les phases déjà complétées (implémentation)
2. Exécute les phases manquantes (discovery, planning)
3. Continue le workflow avec les nouveaux artefacts

Le passage de standard à flash est un no-op — les phases flash sont un sous-ensemble de standard.
