# Modes

> **Ce que vous allez apprendre**
> - Les 3 modes : flash, standard, enterprise
> - Quand utiliser chaque mode
> - Le passage dynamique entre modes

## Comparaison

| Fonctionnalite | Flash | Standard | Enterprise |
|---|---|---|---|
| Temps | < 30 min | 2h - 1 jour | 1 jour - 1 semaine |
| Agents | Lead, Dev | +Analyst, Planner, QA | +DocOps |
| Brief | Non | Oui | Oui |
| Spec | Non | Oui | Oui + ADR |
| Review | Non | Oui | Oui + rapport qualite |
| Approbation humaine | Non | Non | Oui |

## Mode Flash

Pour les changements simples : bug fix, petite feature, refactoring cible.

```bash
$ aiden flash "Fix le bug #123"
```

Quand l'utiliser :
- Moins de 3 fichiers impactes
- Pas de nouvelle API
- Complexite faible (1-2/5)

## Mode Standard

Pour les features moyennes necessitant planification.

```bash
$ aiden standard "Nouvelle API users"
```

Quand l'utiliser :
- 3 a 15 fichiers impactes
- Nouvelle API ou integration
- Complexite moyenne (3-4/5)

## Mode Enterprise

Pour les projets critiques avec compliance.

```bash
$ aiden enterprise "Migration authentification"
```

Quand l'utiliser :
- Plus de 15 fichiers
- Nouvelle architecture
- Compliance requise (RGPD, securite)

### Approbation humaine

En mode enterprise, **chaque phase requiert une approbation humaine** avant de passer a la suivante. A la fin de chaque phase, le CLI affiche :

```
[APPROVAL] Phase "discovery" for "migration-auth" completed.
  Agent: analyst | Tokens: 2340
  Preview: COMPLEXITY: 4 RISKS: - Data loss during migration...
  Approve? (y/n):
```

Si vous repondez `n`, le workflow est avorte avec le statut `aborted`.

Pour bypasser les approbations (par exemple en CI/CD) :

```bash
$ aiden enterprise "Migration auth" --auto-approve
```

> **Attention** : `--auto-approve` desactive toute validation humaine. A utiliser uniquement dans les pipelines automatises.

## Detection automatique du mode

Le Lead analyse la description de la feature avec des heuristiques pour suggerer le mode optimal :

- `"Fix typo in README"` → **flash** (faible complexite, 1 fichier)
- `"Add new REST API with OAuth"` → **standard** (nouvelle API, dependance externe)
- `"Migrate monolith to microservices with RGPD compliance"` → **enterprise** (nouvelle architecture, compliance)

La detection prend aussi en compte le contexte projet (nombre de technologies, architecture).

## Passage dynamique avec re-routing

AIDEN peut changer de mode en cours de route **sans perdre le travail** et en executant les phases manquantes :

```bash
# Demarre en flash, bascule en standard si complexite detectee
$ aiden flash "Export CSV"
AIDEN Lead > Complexite detectee (5 fichiers). Passer en standard ? [Y/n]
```

Quand vous passez de flash a standard, `switchMode()` :
1. Identifie les phases deja completees (implementation)
2. Execute les phases manquantes (discovery, planning)
3. Continue le workflow avec les nouveaux artefacts

Le passage de standard a flash est un no-op — les phases flash sont un sous-ensemble de standard.
