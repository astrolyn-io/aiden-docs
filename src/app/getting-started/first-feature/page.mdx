# Votre premiere feature en mode standard

> Le mode flash, c'etait l'aperitif. Maintenant, on passe aux choses serieuses : **4 agents IA qui collaborent pour livrer une feature complete**. Imaginez une chaine de montage ou chaque poste est tenu par un specialiste — c'est exactement ce qui va se passer sous vos yeux.

Dans ce tutoriel, vous allez suivre en direct le parcours d'une feature a travers les 4 agents du mode standard : **Analyst**, **Planner**, **Dev** et **QA**. Chacun produit des artefacts, chacun est soumis a un checkpoint douanier (quality gate), et le tout s'enchaine comme une chaine de montage bien huilee.

## Le scenario : ajouter GET /health a une API Express

Vous avez une API Express. Vous voulez un endpoint `/health` qui verifie la connexion DB. Une seule commande suffit pour lancer les 4 agents :

```bash
$ aiden standard "Ajouter un endpoint GET /health avec vérification connexion DB"
```

La chaine de montage demarre. Voyons ce qui se passe a chaque poste.

---

## Phase 1 — Decouverte : l'Analyst prend le pouls du projet

Premier sur la chaine, l'Analyst. Son role : scanner le code existant, comprendre le terrain, et produire un diagnostic structure. Il ne code rien — il observe et rapporte.

```
Phase: discovery, Agent: analyst, Provider: claude-code
```

L'Analyst rend son verdict :
- **Complexite** : 2/5 (endpoint simple avec une verification)
- **Risques** : dependance DB non mockee dans les tests
- **Questions** : format de reponse ? verifier la connexion DB ou juste le serveur ?

Il depose son rapport dans `.aiden/features/ajouter-un-endpoint-get-health/brief.md` :

```markdown
---
type: brief
feature: ajouter-un-endpoint-get-health
version: 1
author: analyst
status: draft
---
## Analyse
Endpoint simple retournant le statut du serveur et de la connexion DB.
Complexite: 2/5. Risques: mock DB pour tests.
```

> **Checkpoint douanier (quality gate)** : Si l'Analyst ne detecte pas de complexite ou retourne un contenu vide, la chaine s'arrete. Pas question d'envoyer un dossier vide au Planner — le workflow est bloque jusqu'a resolution.

Le brief est valide. Il passe au poste suivant.

---

## Phase 2 — Planification : le Planner dessine le plan de bataille

Le Planner recoit le brief de l'Analyst et le transforme en specification actionnable. Il decompose le travail en stories avec des criteres d'acceptation precis — exactement comme le ferait un tech lead humain.

```
Phase: planning, Agent: planner, Provider: claude-code
```

Artefacts generes :
- `spec.md` — specification en 1-2 pages
- `stories/001-endpoint-health.md` — story avec criteres d'acceptation
- `stories/002-tests-health.md` — story pour les tests

Voici a quoi ressemble une story :

```markdown
### Story 1: Endpoint /health
Implementer GET /health retournant {status: "healthy", db: "connected"}.

Acceptance criteria:
- GET /health retourne 200 avec JSON
- Si la DB est down, retourne {status: "degraded", db: "disconnected"}
- Le endpoint est accessible sans authentification
```

> **Checkpoint douanier (quality gate)** : Si le Planner ne decompose pas en stories, le workflow est bloque. Pas de plan, pas d'implementation.

Les stories sont pretes, les criteres d'acceptation sont clairs. Le Dev sait exactement quoi construire.

---

## Phase 3 — Implementation : le Dev entre en scene

C'est le moment ou le code est ecrit. Le Dev recoit tout le contexte accumule — brief, spec, stories — et se met au travail. Il ne part pas a l'aveugle : il a la vision complete de ce que les agents precedents ont prepare.

```
Phase: implementation, Agent: dev, Provider: gemini
```

Le Dev suit un processus methodique :
1. Lit les stories et le code existant
2. Implemente le code
3. Ecrit les tests
4. Fait un self-review

Output structure : `filesChanged`, `testsGenerated`.

> **Guardrails** : Si plus de 10 fichiers sont modifies, le circuit breaker intervient — c'est le signal que quelque chose derape. Toutes les 5 etapes, un rappel d'objectif est injecte pour garder le cap. La chaine de montage a ses propres mecanismes de securite.

Le code est ecrit, les tests sont la. Mais avant de declarer victoire, il reste le dernier poste — et pas le moindre.

---

## Phase 4 — Validation : le QA passe tout au crible

Le QA recoit TOUT le contexte : brief, spec, stories, et le code du Dev. Il ne se contente pas d'un coup d'oeil — il verifie explicitement chaque critere d'acceptation, un par un.

```
Phase: validation, Agent: qa, Provider: claude-code
```

Le QA produit un rapport detaille :

```
--- ACCEPTANCE CRITERIA TO VERIFY ---
Story: Endpoint /health
  - GET /health retourne 200 avec JSON
  - Si la DB est down, retourne {status: "degraded"}
  - Accessible sans authentification

CRITERIA_RESULTS:
Endpoint /health / GET retourne 200: PASS
Endpoint /health / DB down: PASS
Endpoint /health / Sans auth: PASS

SCORE: 8/10
PASSED: yes
```

Output structure : `passed`, `score`, `issues`, `criteriaResults`.

### Et si le QA echoue ?

Si `passed = false`, l'orchestrateur ne panique pas. Il lance une **boucle de correction** — comme un pair programming ou le QA pointe les problemes et le Dev corrige en temps reel :

1. Le Dev recoit le feedback du QA et corrige
2. Le QA re-review le nouveau code
3. Maximum 2 retries. Si toujours en echec, le workflow continue avec un warning.

C'est cette boucle de feedback qui fait la difference entre un outil qui genere du code et un framework qui **livre de la qualite**.

## Et apres ? Le mode enterprise

Le mode standard convient a la majorite des features. Mais pour les projets critiques — migration d'authentification, conformite reglementaire, refactoring majeur — il existe un cran au-dessus.

Le mode enterprise ajoute :
- **Approbation humaine** entre chaque phase (vous validez avant de continuer)
- **DocOps** qui genere la documentation et verifie la conformite
- **Quality report** complet

```bash
$ aiden enterprise "Migration authentification" --auto-approve
```

Sans `--auto-approve`, chaque phase demande votre validation avant de continuer.
