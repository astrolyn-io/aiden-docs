# Première feature en mode standard

> **Ce que vous allez apprendre**
> - Le workflow standard complet : Analyst, Planner, Dev, QA
> - Les artefacts générés à chaque phase
> - Que faire si le QA échoue

## Scénario : ajouter GET /health à une API Express

```bash
$ aiden standard "Ajouter un endpoint GET /health avec vérification connexion DB"
```

## Phase 1 — Découverte (Analyst)

L'Analyst scanne le code existant et produit une analyse structurée :

```
Phase: discovery, Agent: analyst, Provider: claude-code
```

Output structuré :
- **Complexité** : 2/5 (endpoint simple avec une vérification)
- **Risques** : dépendance DB non mockée dans les tests
- **Questions** : format de réponse ? vérifier la connexion DB ou juste le serveur ?

Artefact généré : `.aiden/features/ajouter-un-endpoint-get-health/brief.md`

```markdown
---
type: brief
feature: ajouter-un-endpoint-get-health
version: 1
author: analyst
status: draft
---
## Analyse
Endpoint simple retournant le statut du serveur et de la connexion DB.
Complexite: 2/5. Risques: mock DB pour tests.
```

> **Quality gate** : Si l'Analyst ne détecte pas de complexité ou retourne un contenu vide, le workflow est bloqué.

## Phase 2 — Planification (Planner)

Le Planner reçoit le brief de l'Analyst et crée une spec avec des stories :

```
Phase: planning, Agent: planner, Provider: claude-code
```

Artefacts générés :
- `spec.md` — spécification en 1-2 pages
- `stories/001-endpoint-health.md` — story avec critères d'acceptation
- `stories/002-tests-health.md` — story pour les tests

Exemple de story :

```markdown
### Story 1: Endpoint /health
Implementer GET /health retournant {status: "healthy", db: "connected"}.

Acceptance criteria:
- GET /health retourne 200 avec JSON
- Si la DB est down, retourne {status: "degraded", db: "disconnected"}
- Le endpoint est accessible sans authentification
```

> **Quality gate** : Si le Planner ne décompose pas en stories, le workflow est bloqué.

## Phase 3 — Implémentation (Dev)

Le Dev reçoit le brief, la spec et les stories. Il implémente chaque story :

```
Phase: implementation, Agent: dev, Provider: gemini
```

Le Dev :
1. Lit les stories et le code existant
2. Implémente le code
3. Écrit les tests
4. Fait un self-review

Output structuré : `filesChanged`, `testsGenerated`.

> **Guardrails** : Si plus de 10 fichiers sont modifiés, le circuit breaker intervient. Toutes les 5 étapes, un rappel d'objectif est injecté.

## Phase 4 — Validation (QA)

Le QA reçoit TOUT le contexte : brief, spec, stories, code du Dev :

```
Phase: validation, Agent: qa, Provider: claude-code
```

Le QA vérifie explicitement chaque critère d'acceptation :

```
--- ACCEPTANCE CRITERIA TO VERIFY ---
Story: Endpoint /health
  - GET /health retourne 200 avec JSON
  - Si la DB est down, retourne {status: "degraded"}
  - Accessible sans authentification

CRITERIA_RESULTS:
Endpoint /health / GET retourne 200: PASS
Endpoint /health / DB down: PASS
Endpoint /health / Sans auth: PASS

SCORE: 8/10
PASSED: yes
```

Output structuré : `passed`, `score`, `issues`, `criteriaResults`.

### Si le QA échoue

Si `passed = false`, l'orchestrateur lance un **feedback loop** :

1. Le Dev reçoit le feedback du QA et corrige
2. Le QA re-review le nouveau code
3. Maximum 2 retries. Si toujours en échec, le workflow continue avec un warning.

## Passer en enterprise

Pour les projets critiques, le mode enterprise ajoute :
- **Approbation humaine** entre chaque phase
- **DocOps** qui génère la documentation et vérifie la conformité
- **Quality report** complet

```bash
$ aiden enterprise "Migration authentification" --auto-approve
```

Sans `--auto-approve`, chaque phase demande votre validation avant de continuer.
